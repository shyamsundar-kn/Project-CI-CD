# CI-CD-GitOps

A production-style CI/CD pipeline built to automate the entire software delivery lifecycle from source code to a live deployment on AWS EKS. This project demonstrates how modern DevOps practices come together: continuous integration with Jenkins, container security scanning with Trivy, code quality analysis with SonarQube, image management on Amazon ECR, and GitOps-based continuous delivery with Argo CD.

---

## Architecture

```
Developer pushes code
        │
        ▼
  GitHub Repository
  (CI-CD-Gitops)
        │
        ▼
  Jenkins Pipeline
        │
        ├──► Checkout Code
        │
        ├──► Generate Image Tag
        │       (BUILD_NUMBER-git-short-hash)
        │
        ├──► SonarQube Analysis
        │       (Static code analysis & quality gate)
        │
        ├──► Docker Build
        │       (Builds and tags the image)
        │
        ├──► Trivy Image Scan
        │       (CVE vulnerability scan → report.txt)
        │
        ├──► Push to Amazon ECR
        │       (Pushes versioned + latest tag)
        │
        └──► Update GitOps Repository
                (Updates deployment.yaml with new image tag)
                        │
                        ▼
              GitHub Repository
              (CI-CD-dest-repo)
                        │
                        ▼
                   Argo CD
              (Detects change, syncs automatically)
                        │
                        ▼
                 AWS EKS Cluster
              (Rolling deployment applied)
```

## How It Works

Every time a developer pushes code to the main branch, Jenkins automatically triggers the pipeline. Here is what happens at each step:

**1. Checkout**
Jenkins pulls the latest code from the GitHub source repository using stored Git credentials.

**2. Image Tagging**
A unique image tag is generated by combining the Jenkins build number with the short Git commit hash. For example `42-a3f9c1b`. This ensures every build is fully traceable back to a specific commit.

**3. SonarQube Analysis**
The source code is scanned using SonarQube to check for bugs, code smells, and security vulnerabilities. The scan uses the configuration defined in `sonar-project.properties`.

**4. Docker Build**
The application is packaged into a Docker image using the `Dockerfile`. The image is tagged with both the unique version tag and `latest`.

**5. Trivy Image Scan**
The built image is scanned for known CVEs using Trivy. Results are saved to `report.txt` in the Jenkins workspace for review.

**6. Push to Amazon ECR**
Jenkins authenticates with AWS using stored credentials, logs into Amazon ECR, and pushes both the versioned and latest tags to the container registry.

**7. GitOps Update**
Jenkins clones the separate GitOps repository, updates the image tag in `k8s/deployment.yaml` using `sed`, commits the change, and pushes it back to GitHub.

**8. Argo CD Sync**
Argo CD continuously monitors the GitOps repository. When it detects the updated `deployment.yaml`, it automatically syncs the change and triggers a rolling deployment on the EKS cluster with zero manual intervention.

## Pipeline Stages

| Stage | Tool | Description |
|---|---|---|
| Checkout | Jenkins / Git | Clones source code from GitHub |
| Set Image Tag | Shell / Git | Generates `BUILD_NUMBER-short-hash` tag |
| SonarQube Analysis | SonarQube + sonar-scanner | Static analysis and quality gate check |
| Docker Build | Docker | Builds and tags the container image |
| Trivy Image Scan | Trivy | Scans image for CVEs, outputs `report.txt` |
| Push to ECR | AWS CLI + Docker | Pushes versioned and latest image to ECR |
| Update GitOps Repo | Git + Shell | Patches `deployment.yaml` and commits the new tag |

## Setup & Configuration

### Prerequisites

Before running the pipeline, make sure the following are in place:

- Jenkins instance with these plugins installed: Git, Pipeline, SonarQube Scanner, AWS Credentials, Amazon ECR
- Docker installed and running on the Jenkins agent
- Trivy installed on the Jenkins agent and available on `$PATH`
- sonar-scanner installed at `/opt/sonar-scanner/bin/`
- A SonarQube server configured and accessible from Jenkins
- An Amazon ECR repository created in your AWS account
- An EKS cluster running with Argo CD installed
- A separate GitOps repository with `k8s/deployment.yaml`

### Step 1: Clone the Repository

```bash
git clone https://github.com/<your-username>/CI-CD-Gitops.git
cd CI-CD-Gitops
```

### Step 2: Run the Application Locally

```bash
pip install -r requirements.txt
uvicorn main:app --reload
```

Visit `http://127.0.0.1:8000` in your browser.

### Step 3: Build and Run with Docker

```bash
docker build -t lwm-app .
docker run -p 80:80 lwm-app
```

Visit `http://localhost` in your browser.

### Step 4: Add Jenkins Credentials

Go to Jenkins → Manage Jenkins → Credentials and add the following:

| Credential ID | Type | Purpose |
|---|---|---|
| `git-cred` | Username / Password | GitHub repository access |
| `sonar-cred` | Secret Text | SonarQube authentication token |
| `aws-cred` | AWS Credentials | ECR login and image push |

### Step 5: Update the Jenkinsfile

Open `Jenkinsfile` and replace the placeholder values in the `environment` block:

```groovy
environment {
    SONARQUBE_SERVER = 'SonarQubeServer'
    IMAGE_NAME       = 'alphaimage'
    ECR_REPO         = '<account-id>.dkr.ecr.<region>.amazonaws.com/<repo-name>'
    AWS_REGION       = 'ap-south-1'
    DEPLOYMENT_REPO  = 'https://github.com/<your-username>/CI-CD-dest-repo.git'
}
```

Also update the Git repository URL inside the `Checkout` stage to point to your fork.

### Step 6: Create the Jenkins Pipeline Job

- Open Jenkins and click **New Item**
- Select **Pipeline** and give it a name
- Under **Pipeline**, set the definition to **Pipeline script from SCM**
- Set SCM to **Git** and enter your repository URL
- Set the branch to `main`
- Save and click **Build Now**

Jenkins will automatically detect and execute the `Jenkinsfile`.

## Key Design Decisions

- **Immutable image tags**: every build gets a unique `BUILD_NUMBER-commit-hash` tag so deployments are always traceable and rollbacks are straightforward.
- **GitOps separation**: the application source code and the Kubernetes manifests live in separate repositories. This keeps deployment history clean and auditable independently of code history.
- **Security scanning in pipeline**: Trivy runs on every build before the image is pushed, ensuring vulnerabilities are caught early in the process.
- **Zero-touch deployment**: once code is pushed, no manual steps are needed. Argo CD handles the final sync to the cluster automatically.
